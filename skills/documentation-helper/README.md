# Documentation Helper Skill

> Help write clear, comprehensive, and maintainable technical documentation.

## Overview

This skill provides a structured framework for writing high-quality documentation. It helps create:
- **API Documentation** - Clear interface descriptions with examples
- **Code Comments** - Meaningful annotations that explain "why" not "what"
- **README Files** - Project overviews that get users started quickly
- **Architecture Docs** - System design and component documentation

## Quick Start

Simply ask for documentation help:

```
Help me document this function
```

```
Write documentation for the UserAuth module
```

```
Create a README for this project
```

```
Improve the documentation in this file
```

**More specific examples:**

```
Document this API endpoint with parameters and return types
```

```
Add JSDoc comments to this TypeScript file
```

```
Write a migration guide for the v2 to v3 API changes
```

```
Create architecture documentation for the payment service
```

## Documentation Types

| Type | Purpose | Audience |
|------|---------|----------|
| **API Docs** | Describe interfaces, parameters, return values | Developers |
| **Code Comments** | Explain complex logic, design decisions | Developers |
| **README** | Project overview, setup, usage | New users |
| **Architecture Docs** | System design, components, data flow | Developers |

## Core Principles

| Principle | Description |
|-----------|-------------|
| **Clarity** | Simple, direct language; avoid unnecessary jargon |
| **Completeness** | Document all public APIs with examples |
| **Consistency** | Use uniform terminology and style |
| **Accuracy** | Keep docs synchronized with code |

## Common Tasks

| Task | Approach |
|------|----------|
| Document a new API | Use API template with params/returns/errors |
| Add code comments | Focus on WHY, not WHAT |
| Write a README | Follow standard structure: Overview, Installation, Usage, API |
| Update existing docs | Review for accuracy, add missing info |
| Create examples | Use realistic scenarios, test before including |
